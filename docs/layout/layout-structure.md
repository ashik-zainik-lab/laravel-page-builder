# Layout Structure

Layout data is stored within the page JSON file, separate from the primary content sections.

## JSON Representation

In a file like `pages/about.json`, the layout is organized into `header` and `footer` zones. This allows the editor to manage top and bottom structural elements separately from the main content.

```json
{
  "layout": {
    "type": "page",
    "header": {
      "sections": {
        "header": {
          "type": "header",
          "disabled": false,
          "settings": { "sticky": true },
          "blocks": {},
          "order": []
        }
      },
      "order": ["header"]
    },
    "footer": {
      "sections": {
        "footer": {
          "type": "footer",
          "settings": { "copyright": "2024" }
        }
      },
      "order": ["footer"]
    }
  }
}
```

## How Layout JSON is Created

The layout structure is automatically generated by the `LayoutParser` when a new page is created or when the theme defaults are loaded.

### 1. Directive Scanning

The parser scans your layout Blade file (e.g., `layouts/page.blade.php`) for `@sections('key')` directives.

### 2. Zone Partitioning

The parser uses the `@yield('content')` directive as a boundary:

- All `@sections` found **before** `@yield('content')` are placed in the **header** zone.
- All `@sections` found **after** `@yield('content')` are placed in the **footer** zone.

### 3. Default Resolution

For each section key found, the parser:

- Sets the section `type` equal to its `key`.
- Normalizes the structure with `disabled`, `settings`, `blocks`, and `order` attributes.
- This ensures that even if a page JSON has no stored layout data, it can still render with the theme's default structure.

## Section Key Uniqueness

In layout files, each `@sections('key')` must use a **unique key** within the entire layout.

Unlike body sections, you **cannot use the same section key multiple times** in a layout. This is because:

1. **One-to-One Mapping**: Each key maps to a single entry in the `layout.header` or `layout.footer` JSON object.
2. **Settings Collision**: Using the same key twice would cause both instances to share the exact same settings and state, which is usually unintended for layout structural elements.

If you need multiple instances of the same section _type_ in a layout (e.g., two different sub-headers), you should give them unique keys (e.g., `header-main` and `header-top`) and ensure your theme is configured to resolve those keys to the correct templates.

## Layout Blade Files

Your layout Blade file (e.g., `theme/views/layouts/page.blade.php`) should use the `@sections` directive to render these zones:

```blade
<!DOCTYPE html>
<html>
<head>
    <title>@yield('title')</title>
</head>
<body>
    @sections('header')

    <main>
        @yield('content')
    </main>

    @sections('footer')
</body>
</html>
```
